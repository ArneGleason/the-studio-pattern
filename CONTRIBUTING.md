# Contributing

Thank you for helping the Studio Pattern evolve.

This repository is meant to be a canonical, evidence-based home for a lightweight working method. Contributions are most useful when they come from real project use: a handoff that worked, a review that found a memory gap, a command that agents kept rediscovering, or a local adaptation that made a future session easier.

## Responsible Use

The Studio Pattern is intended to support careful human direction, better project memory, clearer review, and healthier collaboration between people and AI agents.

Please do not use or extend this pattern to coordinate harmful work, including deception, harassment, credential theft, malware, evasion of accountability, exploitative labor practices, abusive surveillance, or violations of human rights.

This section is a project norm, not an additional license restriction. The legal license is CC BY-SA 4.0; the community expectation is that people use the pattern with care.

## Pattern Lineage

If you adopt the pattern in another project, please record where it came from.

Suggested note:

```md
Studio Pattern: <version/date/commit>
Source: https://github.com/ArneGleason/the-studio-pattern
License: CC BY-SA 4.0
Local adaptation: <what changed for this project>
```

If the pattern helps, fails, or mutates in an interesting way, consider contributing a short case study or opening an issue describing what happened. The lineage is part of the research value of this project.

## Good Contributions

Useful contributions include:

- clearer templates,
- field-tested workflow refinements,
- case studies from adopting projects,
- review protocol improvements,
- better suspend/resume practices,
- lessons learned from failed handoffs,
- language that makes the pattern easier for a new agent or human to understand.

Avoid adding process because it feels tidy. Add process when it prevents a real repeated mistake or makes handoff measurably easier.

## Evidence Format

When proposing a change, include:

- the source project or situation,
- what went wrong or worked well,
- which existing memory surface helped or failed,
- the smallest reusable change,
- any tradeoff the canonical pattern should acknowledge.

## Case Studies

Case studies should separate verified evidence from interpretation.

Recommended structure:

```md
# <Project> Case Study

## Verified Files

- `<path>`

## What This Shows

<facts observed locally or from linked public evidence>

## Pattern Lessons

- <lesson>

## Gaps Observed

- <gap>

## Influence On The Canonical Pattern

- <possible reusable change>
```

## Pull Requests

For a pull request:

1. Keep the change focused.
2. Update `CHANGELOG.md` when the canonical pattern changes.
3. Update templates when the desired behavior should be copied into new projects.
4. Update docs when the change is explanatory.
5. Add or update a case study when the change comes from field evidence.
